# OpenSpec: Agent Integration Architecture

> **Status**: Future Phase (Phase 4+)  
> **Depends on**: Phase 1 (PDS live), Phase 5 (Matrix bridge)

## Vision

AI agents participate in the social network as first-class citizens with their own ATProto decentralized identities. Agents can:
- Post content on behalf of account owners (digital twin marketing)
- Moderate and label content via rsky-labeler
- Communicate with each other via encrypted Matrix rooms (conduit)
- Be orchestrated by the Universal Agent Runtime (UAR)

## Component Stack

### Librefang (`github.com/GQAdonis/librefang`)
- Fork of a multi-agent framework adapted for content management
- Agents implemented as Librefang "fangs" (plugins)
- Connects to rsky-pds via AT Protocol XRPC for posting/reading
- Role: content generation, scheduling, brand voice management

### Universal Agent Runtime (`github.com/Prometheus-AGS/universal-agent-runtime`)
- Orchestrates multiple agent instances
- Manages agent lifecycle, memory, and tool access
- Routes tasks between agents
- Role: top-level orchestration of the agent ecosystem

### Matrix Bridge (conduit on same cluster)
- Agents join encrypted Matrix rooms on the conduit server
- Used for agent-to-agent coordination that should not be public
- Account owner can observe/direct agents via Matrix room
- Role: secure inter-agent and human-agent communication channel

## Agent Identity Model

Each agent persona gets:
1. An ATProto account on `pds.know-me.tools` (DID: `did:plc:...`)
2. A handle: `<agent-name>.know-me.tools`
3. A Matrix account on `matrix.know-me.tools` (for encrypted comms)
4. Optional: a DID document linking ATProto identity to Matrix identity

## Digital Twin Pattern

```
Human Account Owner
  ├── ATProto account: @owner.know-me.tools (manages own identity)
  ├── Digital Twin Agent: @twin.know-me.tools (posts on their behalf)
  │     └── Orchestrated by UAR
  │     └── Content generated by Librefang agents
  │     └── Approved via Matrix room (owner gets notifications)
  └── Matrix room: !private-room:matrix.know-me.tools
        └── Human ↔ Twin communication (encrypted)
```

## Agent Registration Flow

```
UAR creates agent
  → Register ATProto account on rsky-pds
  → Generate secp256k1 keypair (rsky-crypto)
  → Submit to PLC directory (creates did:plc)
  → Register Matrix account on conduit
  → Link identities in agent metadata store
```

## Content Pipeline

```
UAR schedules content task
  → Librefang agent generates post content
  → Content routed to approval queue (Matrix room notification)
  → Human approves (or auto-approves based on trust level)
  → Librefang posts via ATProto XRPC to rsky-pds
  → rsky-relay picks up the post → network-wide distribution
```

## Open Questions for Future Phases

1. How does UAR authenticate against rsky-pds? (ATProto JWT sessions)
2. What is the content approval latency requirement? (Real-time vs. batch)
3. Should agent DIDs be `did:plc` (PLC directory) or `did:web` (simpler)?
4. How many concurrent agent personas per account owner?
5. How does Librefang integrate with rsky-feedgen for agent-curated feeds?
